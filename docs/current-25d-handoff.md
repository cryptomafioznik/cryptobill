## 1. Куда мы идём

**Цель — техническая игра в духе Gravity Defied с полубоковой 2.5D-проекцией.**
Не однокнопочный раннер. Игрок управляет газом, тормозом и переносом веса; трасса
требует чтения рельефа и выбора техники прохождения, а не удержания одной кнопки.

**Полноценное дорогое 3D отвергнуто окончательно** (заморожено 2026-07-09, `R3D_ON=false`).
Причины: вес, нестабильность, потеря УТП. 2.5D — это и есть продукт, а не ступень к 3D.

**Райдер рисуется гибридно:** художественная основа — существующие v4-PNG, кодом
добавлено только то, чего запечённый PNG не умеет (переосвещение по эпохам,
near/far как одно тело в тени, внешний рим). Чисто процедурный райдер пробовали —
отвергнут (см. §7).

## 3. Архитектура: события контакта

**Граница ответственности.** Физика fixed-step (`loop()`: `physAcc += slowTS;
while(physAcc>=1) update();`), поэтому касание определяется **ровно один раз** —
в `update()`, в цикле по колёсам, до применения сил. Там эмитится read-only событие.

Анимация **потребляет** события и **не переоткрывает** столкновение из render-сэмплов.
Раньше она это делала, и `tContact` расходился на 21 мс, а число событий было 2/2/1
при 30/60/120 FPS.

Поле события: `id`, `simTime`, `wheel`, `contactPoint`, `surfaceNormal`,
`preImpactVelocity`, `normalImpactSpeed`, `airDuration`, `gapPrev/gapCur`.

### Три идентификатора — не путать
- **`contactEventId`** — отдельное физическое касание колеса
- **`airEpisodeId`** — один воздушный эпизод (инкремент при подтверждении воздуха)
- **`landingId`** — одна визуальная реакция на посадку

Оба колеса одного воздушного эпизода дают **один** `landingId`. По каждому
квалифицированному контакту фиксируется решение: `OPEN_LANDING` / `ATTACH_TO_LANDING` /
`UPDATE_IMPULSE` / `IGNORE_ACTIVE_EPISODE` / `REJECT` с причиной.

**Landing envelope стартует от `tContact` события, а не от снятия air-латча.**
Air-латч — только классификатор земля/воздух, источником времени посадки он не является.

### Инварианты (проверяются в рантайме, `window.__rigAssert()`)
- rebound никогда не возникает до compression для того же `landingId`
- один `contactEventId` не потребляется дважды
- `eventAge` не отрицателен
- огибающая не стартует без потреблённого события

### Прочее из Б2
- вся временная база в секундах, лимиты в °/с (210 обычный / 360 hard)
- базовая поза — явный time-based blend с **субкадровой** меткой перехода
  (`crossOffset = threshold − prevAccum`), не рекурсивный `smd()`
- `neutral` разрешён только при устойчивом контакте обоих колёс (50 мс)
- `departurePending`: при потере обоих контактов до подтверждения воздуха
  удерживается предыдущая наземная поза
- путь нагрузки: только подвеска, common → таз Y, differential → таз X/торс,
  от отклонения к медленной базе 1.2 с; локти/колени следуют из таза через IK

## 4. Известные ограничения (backlog)

### B2-KNOWN-01 — лишний wheelie при 120 FPS
При 120 FPS возникает лишний `wheelie` длительностью 25 мс около `t≈2.042`.
При 30 и 60 FPS отсутствует. Из-за него последовательность переходов уезжает
и совпадает 4 из 20 (при 30 vs 60 — 20/20).

**Вернуться только если** проявится визуально на реальной трассе либо повлияет на управление.
Правильное исправление — разделение настоящего wheelie и crest-unload по
замеренным распределениям (`rearContactTime`, `frontAirTime`, `frontGap`, `pitch`,
`angularVelocity`), а не общий debounce.

### B2-KNOWN-02 — активность pose cap
Срабатывает 18 из 430 кадров = 4.19% (цель <1%), максимальная серия 8 кадров (цель ≤2).
14 из 18 приходятся на `elbowF`, преимущественно в воздушных состояниях
(descend 8, ascend 3, apex 2). **На hard landing срабатываний нет.**

**Вернуться, если** дальняя рука визуально щёлкает, зависает или деформируется
в вертикальном срезе. Предел **не повышать** — чинить цель или длительность
blendа дальней руки.

## 5. Как запускать и тестировать

Локальный сервер (`.claude/launch.json`, конфигурация `static`, порт 8123):

```bash
python3 -m http.server 8123
```

URL рига:

```
http://localhost:8123/toys/chartrider.html?render=game&camera=side&rig=cutout&rider=hybrid
```

### Режимы райдера
- `?rider=png` — исходный контрольный (PNG как есть)
- `?rider=hybrid` — **основной кандидат**, используется для приёмки
- `?rider=code` — WIP-эксперимент, отвергнут как основной, не полировать

Дополнительно: `&rigDebug=1` — панель метрик, `&rigSilhouette=1` — серый байк / чёрный райдер.

### Тесты из консоли браузера

```
window.__cutSelfCheck()      // 12 regression gates → {passed:"12/12"}
window.__rigAssert()         // нарушения инвариантов landing-конвейера
window.__capReport()         // статистика ограничителя
window.__capSelfCheck        // см. __cutSelfCheck
```

### Оснастка воспроизведения
- `window.__rigDt` — явный шаг для реплея / FPS-теста
- `window.__rigStateReset()` — **обязателен** перед каждым прогоном
- `window.__rigEventFeed` — подать канонический поток событий вместо `_cev`
- `window.__cevLog` — массив для записи полного потока эмиссии
- `window.__poseSnap / __poseReplay / __poseDiff` — побитовая сверка позы
- `window.__setDPR(d)` — настоящий DPR (игровой `resize()` зажимает до 2)
- `__rgcRun(N, mod, vth, onFrame)` — прогон: `mod=0` газ в пол, `mod=5, vth=2.2` дозированный

**Ловушка:** в production `update()` инкрементирует `_simT` **до** вызова рига.
Реплей обязан стартовать с `_simT = t + SIM_DT`, иначе отстанет на один physics tick
(это давало ложную аномалию «compression=0» и стоило трёх ходов разбора).

## 7. Запрещённые направления и отвергнутые гипотезы

**Не повторять:**
- полноценное 3D — заморожено, убивало УТП
- фото-фоны композитом — провалилось трижды, мир строго процедурный
- чисто процедурный (code-only) райдер — ноги сливаются в массу, торс плоский,
  шлем теряет форму; PNG-силуэт на 45 px честно лучше
- изгибание существующего рендера под другой ракурс — провал b254–b258;
  другая проекция = другой рендерер
- общий minimum-state-duration для всех состояний — задерживает настоящие
  takeoff/landing/air
- повышение предела cap вместо исправления источника

**Гипотезы, проверенные и опровергнутые замером** (не возвращаться без новых данных):
- «расхождение 30/120 FPS даёт покадровая пружина рамы» — перевели во время, не помогло
- «расхождение даёт однокадровый дребезг состояний» — исключили, не помогло
- «расхождение — артефакт интерполяции на несуществующие кадры» — на нативных
  timestamps ошибка сохранилась
- «аномалия compression=0 — дефект контроллера» — оказалась ошибкой оснастки
  (ноль времени реплея)

**Дважды ошибался в измерении, а не в коде:** усреднял максимум в профайлере
(`worst = avg` вместо `max`) и сопоставлял переходы по имени состояния, а не по
порядку/ID. Метрику проверять так же придирчиво, как результат.
