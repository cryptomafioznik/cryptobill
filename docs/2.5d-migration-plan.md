# CRYPTOBILL — план миграции в 2.5D (¾-ракурс)

> Статус на b256 (2026-07-22). Документ отвечает на спеку «полубоковой угол 2.5D» и мапит её на РЕАЛЬНУЮ архитектуру проекта. Обновляется по мере этапов.

## 1. Текущая архитектура (аудит)

- **Стек:** один файл `toys/chartrider.html` (~3100 строк), чистый **Canvas 2D**, без движка/модулей/TS. `index.html` — симлинк. Деплой — GitHub Pages.
- **Игровой цикл:** `loop()` → rAF, fixed-timestep физика + интерполяция (b216), slowmo через дробный `slowLerp`.
- **Физика:** `rigidStep()` — 2D rigid-body (два колеса, подвеска, вес-баланс). Мир одномерен по ходу: `bike.x` = прогресс (worldS), `bike.y` = высота. **Бокового смещения (worldLateral) в симуляции НЕТ и не планируется** — трасса = реальный график цены (USP), физика читает `terrainAt(x)`.
- **Трасса = сегменты УЖЕ:** массив `terr[]` узлов с шагом `STEP=26` (`x,y,open,close,hi,lo` + флаги фич `kick/gap/mega/climb/whoop/mud/haz/coin`). Это и есть требуемые track segments: `s=x/STEP`, `height=y`, `slope` эмерджентно, obstacle refs = флаги. Heading/curvature отсутствуют осознанно (график нельзя гнуть).
- **Камера:** `cam{x,y}` — байк на ~32%W / 52%H, follow-lerp, air-подстройка, зум `_zoomD`. Мир рисуется в `ctx.translate(-cam)`-блоке.
- **HUD/управление:** DOM-кнопки + экранные draw ПОСЛЕ restore — **уже отделены от мира**, проекцией не затрагиваются. Мультитач/safe-area — рабочие (b56+).
- **Параллакс:** небо/дальний скайлайн/город-свечи ×2/пальмы/вода — слои с разными `par`-коэффициентами.
- **Перф:** кэш-спрайты байков, кэш градиентов, без shadowBlur в горячих циклах (b253). Окно отрисовки трассы `i0..i1` = culling.

## 2. Что из спеки УЖЕ реализовано (b254–b255)

| Спека | Реализация |
|---|---|
| Projection25D (world→screen, scale, depth) | `psp3(x,y)→[x',y',s]` + `pspY(y,s)` — лог-перспектива `x'=px+D·ln(1+dx/D)`, схождение к горизонту `_pspHY`, масштаб `s`. Кнобы `PSP={D,KY,RW}` |
| Разделение симуляции и проекции | Физика не тронута вообще; проекция применяется только в draw-пути (вершины ленты, свечи-колонны, кромки, рампы, чипы, хазарды, монеты — позиция+размер×s) |
| Режимы камеры side2d / threeQuarter25d | Тумблер «РАКУРС ¾» в настройках (`PSP_ON`, localStorage `cb_psp`) + query `?camera=side|25d` (b256) |
| Игрок в ¾ (этап 5) | Пробник `drawMachine34`: КРОСС в ¾ — эллипс-цилиндры колёс с бегущими кнобами, обе ноги вилки, руль двумя грипсами, райдер со спины с живыми конечностями, подвеска жива. Гейт `PSP_ON && frame==='cross'` |
| Culling / удаление за камерой | Существующее окно `i0..i1`; psp монотонна — за-камерные не рисуются как и раньше |
| Depth-cue объектов | Монеты/чипы/хазарды масштабируются `×s`; painter's order прежний (мерцаний нет: трасса→фичи→байк) |
| HUD неподвижен | Был отделён изначально |

## 3. Этап b256 (этот коммит) — «дорога-лента + композиция кадра»

- **Дорога с ВИДИМОЙ ШИРИНОЙ**: полутолщина ленты ×`PSP.RW` (1.8) при PSP_ON → у байка дорога широкая, впереди сходится к горизонту (перспектива KY сжимает). Свечи-колонны посажены под новую нижнюю кромку.
- **Композиция как в референсе**: при PSP_ON байк ниже (60%H вместо 52%) и левее (30%W), горизонт выше (`_pspHY=cam.y+H*0.30`) → город/закат в верхней половине, трасса-впереди доминирует.
- **Сильнее угол по умолчанию**: `PSP.D` 1500→1000, `KY` 0.88→0.92.
- **`?camera=side` / `?camera=25d`** — сравнение без настроек (не персистится).

## 4. Осознанные отступления от спеки (с причинами)

- **Нет `src/`-структуры / TS** — проект однофайловый по стилю; модули = помеченные секции (`psp*`, `drawMachine34`, `drawTrackReal`). Дробить файл в середине визуального эпика = риск без пользы.
- **Нет heading/curvature (изгибов трассы)** — трасса = реальный график (USP). Изгиб был бы чистой косметикой, ломающей чтение графика. Отклонено.
- **worldLateral игрока = 0** — управление 1D по ходу; боковых полос нет в геймдизайне.
- **Тени под перспективой**: контакт-тень байка есть; отдельные эллипс-тени монет — в полировку (этап 10).
- **Тест-инфраструктуры нет** — проверка = headless-сим реального движка (browser JS: подавить rAF → `update()`-прогон, скрин-сравнение) + `tools/sim.js`. Формальные юнит-тесты для однофайлового canvas — вне бюджета этапа.

## 5. Оставшиеся этапы (порядок)

1. **¾-арт остальных 6 байков + трюк-позы** (после вердикта юзера по пробнику) — самый большой кусок.
2. Параллакс-глубина фона под новый горизонт (масштабные коэффициенты слоёв, дымка сильнее к точке схода).
3. Тени/полировка: эллипс-тени объектов, speed-lines, отражения.
4. Перф-проход на телефоне (FPS-оверлей есть: dbg, клавиша F).
5. Решение по Забегу: ¾ везде или только Отрыв/Карьера (`trackSource!=='ticker'` гейт = 1 строка) — за юзером.

## 6. Критерии готовности (маппинг)

Из 18 пунктов спеки: 1–3, 5–8, 10–16 — выполнены к b256 (¾-игрок = пробник-кросс); 4/9 (глубина+рельеф) — выполнены, калибровка угла за телефоном; 17 (перф) — ждёт замера на устройстве; 18 — этот документ. Пункт «трасса шире внизу экрана» — b256.

## 7. Настройка (кнобы)

- `PSP.D` — глубина перспективы (меньше = круче угол; 1000 дефолт, 650 «сильный», 1500 «мягкий»)
- `PSP.KY` — доля схождения к горизонту (0.92)
- `PSP.RW` — ширина дороги у камеры (1.8)
- `_pspHY` — якорь горизонта (H*0.30)
- камера: `W*0.30 / H*0.60` при PSP_ON (в `update`, ~L1535)
- `?camera=side|25d`, тумблер в настройках, `localStorage cb_psp`

## 8. Риски

- Читаемость графика в Забеге при сильном угле (решение юзера).
- ¾-арт 7 байков × позы — объём; смягчение: пробник-подход, по байку за раз.
- Слишком широкая дорога может съедать высоту рельефа визуально — калибровка RW на телефоне.
