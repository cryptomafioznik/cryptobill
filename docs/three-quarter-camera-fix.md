## ЭТАП 1. Аудит текущей неудачной реализации

### 1. Где хранится старая боковая heightmap
`let terr=[]` (объявление ~стр.84). Узел:
```
terr[i] = { x, y, open, close, hi, lo, up, gap, topY, kick, whoop, mud, drop, climb, mega, coin, haz, ... }
```
- `x` — мировая горизонталь, шаг `STEP=26` (`genX+=STEP`).
- `y` — **экранно-мировая вертикаль** (цена свечи, нормализованная в Y; МЕНЬШЕ = ВЫШЕ, ось вниз по канвасу). Это heightmap `y = f(x)`.
- `terrainAt(x)` (стр.228) → `{y, ang}` линейной интерполяцией между узлами. `ang=atan2(dy,STEP)` = наклон в плоскости x–y.
- **Z-оси нет. lateral нет.** Трасса — одномерный профиль высоты вдоль `x`.

### 2. Как из неё строится широкая дорожная полоса (то, что забраковано)
Два экранных рендерера:
- `drawTrackReal(i0,i1)` (стр.~1722, side/legacy) — косоугольная лента: низ = верх + константный вектор `(TRV_X,TDECK)` в экранных координатах.
- `drawTrack25D(i0,i1)` (стр.~1750, b258) — **экранно-пространственный варп**: центр `Xc=VPX·W+(CAMX−VPX)·W·s`, края `Xc∓HW·W·s`, где `s=F/(F+d)`, `d=x−bike.x`. Это ширина, «нарисованная» в экране, **без мировой Z-глубины**. Ровно запрещённый паттерн.

### 3. Как вычисляется экранная позиция мотоцикла
Через 2D-скролл `cam={x,y}`: `ctx.translate(-cam.x,-cam.y)` → байк рисуется в мировых `(bike.x,bike.y)`, `cam.x=bike.x−W·CAMX`, `cam.y≈bike.y−H·CAMY` (стр.~1541). Плюс `ctx.scale(PSP.BS)` (постоянный множитель героя) и `ctx.scale(zoom)` (кинематограф). **Никакой проекции — только смещение и масштаб.**

### 4. Существует ли настоящая camera/view/projection matrix
**НЕТ.** `cv.getContext('2d')` (стр.68), единственный канвас `#c`. Нет `mat4`, `lookAt`, `perspective`, WebGL. `cam` = 2 скаляра сдвига. (Старый Three.js-слой был вырезан в b251, стр.704.)

### 5. `cameraYaw` — реальный поворот или экранный коэффициент
`cameraYaw` **не существует**. Ближайшее — `PSP.VPX/VPY` (доля экрана точки схода) и `F` (коэффициент сжатия). Это чистые экранные множители, **не углы камеры**.

### 6. Одна ли система перспективы у монет/препятствий
Частично «одна», но НЕ мировая: монеты/чипы/рампы/хазарды зовут `psp3(x,y)` — тот же **экранный** варп. Размер объекта = `×s` (экранный коэффициент), не следствие мировой дистанции через проекцию. lateral у объектов нет (все на осевой линии).

### 7. Что можно отключить только для режима `threeQuarter25d`
Гейт — флаг `PSP_ON` (`cb_psp` / `?camera=25d|side`). Для нового режима в ветке `PSP_ON`:
- НЕ звать `drawTrack25D` / `drawTrackReal` / `drawCandleBodies` / `psp3`-экранный варп — заменить на новый мировой рендер `render25D()`.
- `drawMachine34` (¾-арт кросса, стр.2605) — временно переиспользуем как спрайт-заглушку, позиционируя по мировой проекции (не фикс-масштаб).
- Фон-мир (`drawCityWorld`), HUD, частицы, меню — **остаются 2D-слоями** (совпадает с требованием «HUD отдельным 2D-слоем»). Задний фон рисуется по мировому горизонту камеры.
- Физика/`rigidStep`/`update`/`terrainAt` — **НЕ трогаем** (коллизии остаются в координатах трассы). Только адаптер `x→s`, `terr.y→worldY`.

## Выбор архитектуры (решение — за юзером, см. вопрос в чате)

Проект — чистый Canvas 2D; Three.js был осознанно вырезан (b251: 1.2МБ CDN каждому игроку, тяжело/нестабильно на мобиле, убивал перф/УТП). Спека допускает **чистый Canvas при условии полноценного преобразования мировых X/Y/Z через view/projection matrix**.

**Рекомендация: чистый Canvas + собственные mat4 (view + perspective).** Причины: без CDN/бандла (не повторяем регресс b251), однофайловость, лёгкость на мобиле (дорога ≈ 60–170 квадов/кадр — копейки для CPU), весь существующий 2D-слой (HUD/меню/город/частицы) остаётся нетронутым — ровно как требует спека «HUD отдельным 2D-слоем». Глубина — **painter's algorithm** (дорога монотонна по глубине; объекты сортируются по `s`) вместо попиксельного z-buffer — честно: это не z-buffer, но для сцены «дорога+объекты» корректно и достаточно. Альтернатива Three.js даёт GPU z-buffer, но возвращает вес CDN + композит двух канвасов + разворот замороженного решения на полную стоимость — хуже для этого проекта.

## План реализации (Этапы 2–10)

- **Э2. Координаты.** `s=bike.x` (вдоль осевой), `worldY=(baseY−terr.y)·MZ` (вверх=+Y), `worldX=lateral` (0 на прямой), `worldZ=s`. Прыжок → `airHeight`. `bike.ang` → pitch байка относительно касательной. Касательная `T(s)`, боковой `R(s)=cross(up,T)`, края `P(s)∓R·roadW/2`.
- **Э3. Дорога-mesh.** Поперечные сечения `left(s)/right(s)`, шаг 0.5–2 ед., 80–120 ед. вперёд, painter far→near, UV/пунктир по `s`, без щелей/экранных стен, за камерой не рисуем.
- **Э4. Камера.** `lookAt(eye,target,up)` + `perspective(fov 40–50°,near,far≥150)`. eye = позади (6–10) + сбоку (3.5–6, знак `cameraSideSign`) + выше (3–5.5) относительно локального базиса трассы; target — впереди игрока (look-ahead 12–22); side-angle 25–45°, pitch 15–28°; демпфирование позиции/цели. Прыжок влияет на камеру ≤20–35%.
- **Э5. Байк 3/4.** Прототип: слоёная 3D-конструкция из простых деталей ИЛИ `drawMachine34` как заглушка по мировой позиции `P(s)+R·lateral+N·(wheelR+airHeight)`; yaw=T(s), pitch=уклон+прыжок, roll ограниченный. Ось байка ↔ проекция касательной ≤8°.
- **Э6. Объекты.** Хранят `s/lateral/heightOffset`, идут через ту же камеру; отдельного экранного scale нет; за камерой отбрасываются; препятствия видны ≥2с.
- **Э7. Калибровочная сцена** `?camera=25d&calibration=1`: прямая 120 ед., поперечины/10 ед., центр-пунктир, статичный байк, объекты на 10/20/40, точка схода, стрелки tangent/forward, числа yaw/pitch/FOV/look-ahead. **Кадр правим ЗДЕСЬ до подключения холмов/физики/токенов.**
- **Э8–10.** Адаптер физики → холмы/повороты → HUD/управление → эффекты/оптимизация. Каждый этап — только после прохождения геометрической проверки предыдущего.

## ЭТАП 7 — РЕЗУЛЬТАТ (b259): калибровочная сцена ГОТОВА, все геометрические гейты пройдены

Рендерер `drawTrack25D`-эпохи УДАЛЁН из ¾-пути; построен настоящий world-space пайплайн (см. `chartrider.html`, блок «b259 РЕЖИМ 25D»): `m4perspective` + `m4lookAt` (свои mat4, column-major), `m4xp`→clip, homogeneous divide, near-plane clip (`clipNearPoly`/`seg3`), **painter's НА УРОВНЕ ТРЕУГОЛЬНИКОВ** (`paintTris` сортирует по view-глубине). ЕДИНАЯ проекция: дорога, кубы-объекты, байк (3D-конструкция из боксов+колёс-цилиндров), контакт-тень — всё через `toClip`. Экранные x/y НЕ источник перспективы. URL: `?camera=25d&calibration=1`.

**Калиброванные параметры камеры** (`CAM25`, проходят все гейты на 390×844): `fov 54°, near 0.2, far 240, back 9.8, side 1.5, height 5.5, lookAhead 7.2, targetLift 2.3, sideSign −1, roadW 3.6, bikeS 12`. Камера следует локальному базису трассы (T/R/up), смотрит вперёд+чуть выше игрока.

**Два конфликта спеки (Stage-4 guidance ↔ hard-гейты) — разрешены в пользу измеримых гейтов, честно:**
1. **side-angle 8.7°** (guidance 25–45°). Гейт `центр байка X 28–42%` при байке на осевой + камере сбоку СТРУКТУРНО ограничивает side-angle ~10–13° (при 25° байк уезжает на 10%W = мимо гейта). Сильнее ¾ = только если ослабить гейт bikeCx (байк дальше влево, ~15–25%W). Вид сейчас = ¾-сзади-сверху с лёгким боком.
2. **fov 54°** (guidance 40–50°). Нужен чтобы развязать coupled-гейты `a20up≥25` ∧ `bikeCy≤75` (обе точки на полотне двигаются вместе; при fov≤50 одна на 0.1–0.7 вне диапазона).

**Перф:** 0.46 мс/кадр CPU (~300 треугольников) на маке → на мобиле доля бюджета. Painter's-по-треугольникам даёт корректные перекрытия в этой сцене (near рисуется поверх far). Если на холмах (Э8) painter's начнёт ошибаться — уходим на WebGL (по указанию юзера).

**Коллизии от экранного масштаба НЕ зависят:** физика (`update`/`rigidStep`/`terrainAt`) не тронута, работает в world-координатах трассы (terr.x/terr.y); модуль 25D — только вывод. Калибровка вообще без физики. `grep CAM25|measure25D` в update/rigidStep = 0.

**Ассеты:** временные — 3D-байк = слоёная конструкция из боксов+колёс-цилиндров (`buildBike25`), кубы-объекты (`addTestObj`), тень-октагон. Финальные — нет (следующий этап: low-poly байк/6 моделей + трюк-позы).
