# trialsOblique25d — план (простой oblique 2.5D, БЕЗ настоящего 3D)

Билд на момент аудита: b260. Файл: `toys/chartrider.html` (однофайловый Canvas 2D). Новый режим — `?render=trials25d`; старый `?camera=25d` (mat4-перспектива b259/b260) в новом режиме НЕ используется.

> **⚠ Разворот направления.** b259/b260 = настоящая mat4 view+projection (по прошлому ТЗ). Новое ТЗ ЯВНО запрещает 3D/mat4/perspective camera → строим ПРОСТОЙ oblique. Причина осмысленна: сильная перспектива превращала график в дорогу и убивала УТП-читаемость. Слабый oblique её сохраняет.

## Этап 1 — Аудит

1. **Физическая модель.** `rigidStep()` (стр.1291, rigid-body b68) + `update()` (стр.1369). Полностью 2D: `bike={x,y,vx,vy,ang,angV,onG,wheel,rShift}` (стр.1030). Мир — экранно-мировые x/y.
2. **Профиль трассы.** `terr[]` — массив узлов `{x,y,open,close,hi,lo,up,gap,topY,kick,whoop,mud,drop,climb,mega,...}`, шаг `STEP=26`. `y` = высота (МЕНЬШЕ=ВЫШЕ). `terrainAt(x)→{y,ang}` (стр.228) линейной интерполяцией. `baseY=H*0.6` (стр.1017) — базовая линия.
3. **Контакт колёс.** В `update()`/`rigidStep`: `terrainAt(bike.x−DX)` / `terrainAt(bike.x+DX)` (стр.1385) = высота под задним/передним колесом; наклон опоры `supA`. Контакт-пятна рисуются от `terrainAt(wx).y` (стр.1867).
4. **График→трасса.** `buildTickerTrack(candles)` (стр.244): OHLC-свеча → нормализация в Y → узлы `terr` (та же структура, что `genSeg`). `genSeg()` (стр.136) — процедурная генерация (холмы/гэпы/рампы).
5. **Canvas-renderer.** `draw()` (стр.1824), один `ctx=cv.getContext('2d')`. `drawTrackReal` (боковая лента), `drawCandleBodies`, `drawCityWorld` (synthwave-фон: `drawSun`/`cityPass`/`drawPalms`/`palmSprites`, `skyline.png`, `coin.png`).
6. **Прошлые 2.5D/mat4.** b254–258: `psp3` экранный варп (`drawTrack25D`) — забракован. b259–260: настоящие mat4 (`cam25Build`/`m4perspective`/`m4lookAt`/`paintTris`/`drawCalibration25D`), гейт `?camera=25d`. **В новом режиме НЕ используется** (по ТЗ), но остаётся для сравнения.
7. **Переиспользуем.** `terr`/`terrainAt`/`rigidStep`/`update`/`buildTickerTrack` (физика+трасса, не трогаем); synthwave-фон (`drawSun`/`cityPass`/`drawPalms`/`palmSprites`/`coin.png`); HUD DOM (screen space). Контролы (`down`/`brake`/`leanX`/`jumpReq`, `recompute`).
8. **Текущее управление/физика.** Газ (`down`), тормоз (`brake`), перенос веса (`leanX`→`rShift`), прыжок (`jumpReq`). rigidStep = снаряд над линией; вылет геометрический (b149), баланс на крутых (b232). Не трогаем.

## Отделение физики от визуализации
Физика оперирует `s`(=`bike.x`), `trackHeight(s)`(из `terr.y`), `airHeight`, `bikeAngle`, контакты колёс — как сейчас. Renderer добавляет ТОЛЬКО декоративную `visualDepth(s)` (медленная низкочастотная функция), которая **не участвует в физике/коллизиях/позиции токенов**. Оба края ленты и байк проецируются одной `projOB()`. Инвариант: убери `visualDepth` → трасса и вся физика идентичны side2d.

## Формулы проекции (oblique, без mat4)
```
u = s - cameraS
z = visualDepth(s)                       // декоративная глубина
persp = 1 - clamp(u/uFar,0,1)*perspFade  // слабое уменьшение, perspFade ≤0.15
screenX = anchorX + u*scaleX*persp + z*depthVectorX
screenY = anchorY - (trackHeight(s)-cameraHeight)*scaleY*persp + u*longitudinalTilt + z*depthVectorY
```
Ориентиры: `depthVectorX 18–30`, `depthVectorY −8..−18`, `longitudinalTilt −0.03..−0.08`, `scaleY 0.9–1.15·scaleX`, `perspFade ≤0.15`. Края: `nearDepth=visualDepth(s)−roadVisualWidth/2`, `farDepth=+roadVisualWidth/2`; поверхность = квад `[near_i, near_{i+1}, far_{i+1}, far_i]`. Визуальная касательная: `atan2(project(s+ε)−project(s−ε))`.

## Риски
- Слишком сильная глубина → «дорога/вид сзназад»: держать амплитуду ≤0.3–0.6 ширины, wavelength длинную, perspFade малым.
- Толстая лента → «автомобильная дорога»: ширина поверхности 24–35px, боковая грань ≤8px, без опор/оснований.
- Сглаживание графика → «обычные холмы»: сохранять экстремумы/крутые уклоны 45–70°.
- Читаемость под HUD/на глубине: снизить контраст фона за трассой.

## Этапы

График (реальные данные, экстремумы, проходимость). 7. Токены/эффекты. 8. HUD/оптимизация.

## Критерии приёмки (для calibration, 390×844)
Байк 11–14%H, центр X 28–38%W, впереди ≥60%W, ширина поверхности у игрока 24–35px, near+far neon edge видны, боковая грань ≤8px, ≥3 элемента испытания в кадре, профиль поддерживает ≥60° уклоны, `visualDepth` не в коллизиях, ошибка направления байка vs projected tangent ≤5°, НЕТ дороги/точки схода, НЕТ 3D-renderer, HUD не движется с камерой.
